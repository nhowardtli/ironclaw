 4. Session Establishment and Capability Negotiation 
4.1 Purpose 
VIRP implementations MUST establish a negotiated session before exchanging Observation, Intent, or future protocol-channel messages. 
The session-establishment phase serves four purposes: 
protocol version negotiation
capability negotiation
algorithm negotiation
freshness and session binding via nonce exchange 
This phase converts VIRP from a message-signing format into a stateful trust protocol between an AI control plane and an Observation Node (“O-Node”).
This is consistent with VIRP’s architecture, which separates collection/signing from AI-side consumption. 
 4.2 Session State Machine 
A VIRP session SHALL follow this state machine: 
DISCONNECTED
↓
HELLO_SENT
↓
NEGOTIATED
↓
SESSION_BOUND
↓
ACTIVE
↓
CLOSED

Message-bearing protocol channels such as OBSERVATION and INTENT MUST NOT be exchanged before the session reaches SESSION_BOUND. 
 4.3 Message Types 
The following session-establishment messages are defined: 
HELLO
HELLO_ACK
SESSION_BIND
SESSION_CLOSE 
Future extensions MAY define additional session-control messages. 
 4.4 HELLO 
The HELLO message is sent by the initiating peer (typically the AI-side client) to advertise supported protocol parameters.

 4.7 SESSION_CLOSE 
Either peer MAY terminate a VIRP session with SESSION_CLOSE. 
A SESSION_CLOSE message SHOULD include:Field Type Descriptionmsg_type string Fixed value: "SESSION_CLOSE"session_id string Session being closedreason_code uint16 Numeric close reasontimestamp_ns uint64 Local send time (nanoseconds since epoch) 
After sending or accepting SESSION_CLOSE, the session state becomes CLOSED. 
 4.8 Session-Bound Message Requirement 
All VIRP messages sent after SESSION_BOUND MUST include session_id as part of the authenticated context. 
For protocol version 2 and later, signed Observation messages MUST cryptographically bind at minimum: 
version
session_id
node_id
timestamp_ns
seq_num
channel
tier
device_id
command_hash
payload or payload_hash 
This extends the trust guarantee from payload authenticity to session-scoped, source-scoped, and command-scoped authenticity. 
 4.9 Negotiation Failure 
If no compatible protocol version, channel set, or algorithm exists, the responder MUST reject session establishment with an ERROR message:Field Type Descriptionmsg_type string Fixed value: "ERROR"error_code uint16 Numeric error identifiererror_text string Human-readable reasontimestamp_ns uint64 Timestamp

 4.10 Replay and Freshness Considerations 
The client_nonce, server_nonce, session_id, and monotonically advancing seq_num together provide the basis for replay detection. 
Implementations SHOULD reject: 
duplicate SESSION_BIND messages for an active session
message replays with stale session_id
repeated seq_num values within a session
timestamps outside an implementation-defined freshness window 
 4.11 Minimal Implementation GuidanceA minimal VIRP implementation MAY begin with: 
one supported algorithm: HMAC-SHA256
one required channel: OBSERVATION
one negotiated version: 2 
Such an implementation still MUST perform HELLO → HELLO_ACK → SESSION_BIND before accepting Observation messages. 
 4.12 Rationale 
This handshake model gives VIRP explicit versioning, explicit capability negotiation, explicit session identity, and nonce-based freshness context —
converting VIRP from a local signing library into a stateful, negotiated trust protocol. 
VIRP is not just a signing format — it is a negotiated trust session between an AI control plane and a reality-collecting O-Node. 
 5. Observation Context Binding (v2) 
5.1 Problem 
Version 1 VIRP observations cryptographically bind the observation payload and core header fields (node_id, timestamp_ns, seq_num, channel,
tier, version). However, the device identity and command that produced the observation are not included inside the cryptographic boundary. 
This creates an attribution ambiguity: a valid observation payload could theoretically be reassigned to a different device or command at a higher layer. 
5.2 Requirement 
Observation messages in protocol version 2 MUST cryptographically bind: 
collector identity (node_id)
observation timestamp (timestamp_ns)
sequence number (seq_num)
channel
tier
protocol version
device identity (device_id)
command identity (command_hash)
session identity (session_id)
observation payload 
5.3 Security Guarantee 
Once signed, a v2 observation is a cryptographic commitment to: 
collector identity
+ session context
+ time/order position
+ device identity
+ executed command
+ returned payload

Any reassignment of payload to a different device or command MUST invalidate verification. 
5.4 VersioningVersion Trust Guaranteev1 Payload authenticv2 Payload + context authentic 
Nodes SHOULD accept both versions during transition periods. The negotiated session version (from HELLO/HELLO_ACK) determines which guarantee
applies. 
Public framing: 
V1: Prove the AI didn’t fabricate reality
V2: Prove the reality came from the claimed source, session, and command 
 6. Wire Format v2 
6.1 Observation Header v2 
struct virp_obs_header_v2 {uint8_t version; // VIRP_VERSION_2
uint8_t channel; // VIRP_CHANNEL_OBS
uint8_t tier; // GREEN / YELLOW / RED / BLACK
uint8_t _reserved; // padding, must be zero
  uint64_t node_id; // stable O-Node identity
uint64_t timestamp_ns; // nanoseconds since epoch
uint64_t seq_num; // monotonically increasing per session
  uint8_t session_id[16]; // from SESSION_BIND
uint64_t device_id; // stable device UUID (see §6.3)
  uint8_t command_hash[32]; // SHA-256 of canonical command (see §6.4)
uint32_t payload_len; // length of observation payload in bytes
};

6.2 Signing 
Signed material is the full header concatenated with the payload: 
HMAC_SHA256(
key,
key_len,
&hdr,payload,
payload_len,
signature_out
);

6.3 Canonical Device Identity 
device_id MUST be a stable uint64_t UUID assigned at device onboarding and stored in devices.json. Device display names or hostnames MUST
NOT be used as device_id. Renames MUST NOT change device_id. 
6.4 Canonical Command Hash 
command_hash = SHA-256(canonical_command_string)

Canonicalization rules (applied in order): 
Trim leading and trailing whitespace
Collapse repeated internal spaces to single space
Normalize line endings to \n
Strip CLI prompts and transport-specific wrappers 
Example: " show ip bgp summary\r\n" → "show ip bgp summary" → SHA-256 
Implementations MUST apply identical canonicalization on both the signing side (O-Node) and the verification side (bridge/AI layer). 
6.5 Optional: Collector Type 
Implementations MAY include a collector_type field to bind transport context:Value Transport0x01 SSH0x02 REST API0x03 SNMP0x04 Agent/SDK 
This field is advisory. It MUST be inside the HMAC boundary if present.
